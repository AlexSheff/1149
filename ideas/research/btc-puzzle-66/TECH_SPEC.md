# BTC Puzzle #66 — Technical Research Notes

> *Research notes on solving the Bitcoin Puzzle #66: finding the private key for a known public key within a defined key range.*

---

## 1. Problem Definition

The Bitcoin Puzzle is a series of known public addresses with defined private key ranges:

```
Puzzle #66:
  Target address: 13zb1hQbWVsc2S7ZTZnP2G4undNNpdh5so
  Key range:      [2^65, 2^66 - 1]
  Range size:     ~36.9 quintillion keys (2^65 keys to check)
  Prize:          6.6 BTC (unclaimed)
```

The private key `k` satisfies: `k * G = P` where `G` is the secp256k1 generator point and `P` is the target public key.

## 2. Mathematical Foundation

### secp256k1 Parameters

```
p  = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEFFFFFC2F
n  = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141
Gx = 0x79BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798
Gy = 0x483ADA7726A3C4655DA4FBFC0E1108A8FD17B448A68554199C47D08FFB10D4B8

# Generator point G = (Gx, Gy)
# Curve: y² = x³ + 7 (mod p)
```

### Baby-Step Giant-Step (BSGS) Algorithm

The most tractable approach for bounded key ranges.

```python
import math
from typing import Optional

def bsgs(G, P, n: int, range_start: int, range_end: int) -> Optional[int]:
    """
    Baby-Step Giant-Step for discrete log in [range_start, range_end].
    G: generator point, P: target public key, n: group order
    """
    search_range = range_end - range_start
    m = int(math.ceil(math.sqrt(search_range))) + 1

    # Baby steps: compute and store {j*G: j} for j in [0, m)
    # (shifted: we look for k - range_start, so precompute from 0)
    baby_steps = {}
    point = POINT_AT_INFINITY
    for j in range(m):
        baby_steps[point_to_bytes(point)] = j
        point = point_add(point, G)

    # Giant steps: P - range_start*G - i*m*G for i in [0, m)
    # If match found: k = range_start + j + i*m
    offset = scalar_mult(range_start, G)
    current = point_subtract(P, offset)
    m_step = scalar_mult(m, G)

    for i in range(m):
        key = point_to_bytes(current)
        if key in baby_steps:
            candidate = range_start + baby_steps[key] + i * m
            if candidate <= range_end:
                return candidate
        current = point_subtract(current, m_step)

    return None
```

**Complexity:** O(√range) time, O(√range) space
**For puzzle #66:** O(2^32.5) ≈ 6 billion operations — feasible with GPU acceleration

## 3. GPU Acceleration Strategy

Pure Python BSGS is too slow. Target: CUDA implementation.

```
CPU BSGS speed:    ~1M keys/sec
GPU BSGS (A100):   ~1B keys/sec  
GPU BSGS (4×A100): ~4B keys/sec

Time estimate (4×A100): 2^65 / 4B = ~9,200 years  ← still infeasible alone
```

**Conclusion:** Pure BSGS at range scale is not feasible without additional constraints.

---

## 4. Constraint Reduction Approaches

### Kangaroo Method (Pollard's Kangaroo)

Better for large ranges. Two kangaroos (tame + wild) perform pseudorandom walks that eventually collide.

```
Expected operations: ~2 × √(range_end - range_start)
For puzzle #66: 2 × √(2^65) = 2 × 2^32.5 ≈ 12 billion
```

Still requires GPU. But: distributed kangaroo (many clients) is parallelizable.

### Dormant Wallet Pattern Analysis

Hypothesis: keys generated by early Bitcoin clients had predictable entropy patterns.

Research sources:
- `dormantbitcoin.com` — dormant address patterns
- `btcpuzzle.info` — community research on puzzle structure
- `directory.io` — key space visualization
- `allprivatekeys.com` — reference database

If the puzzle creator used a specific derivation pattern, the effective search space may be smaller.
